# FDR-LOCAL Private Key Exposure in iOS Firmware Ramdisks

## Summary

Apple's FDR (FairPlay Data Recovery) local signing private keys -- both RSA-2048 and ECDSA P-256 -- are embedded in plaintext within libFDR.dylib shipped on every iOS restore ramdisk. These keys are identical across all device models and firmware versions, and are trivially extractable from publicly downloadable IPSW files using only the strings command. The corresponding certificates (CN=FDR-LOCAL-V1 and CN=FDR-LOCAL) are valid until 2032 and are used by libFDR's AMFDRCryptoCreateLocalSigned* family of functions to sign FDR certificates, data signatures, and device identity material during restore operations.

Using identical static private keys across the entire iOS device fleet -- rather than per-device keys derived from the Secure Enclave -- means that extracting one IPSW compromises the FDR local trust chain for every Apple device. An attacker with these keys can invoke AMFDRCryptoCreateLocalSignedCertificateData and AMFDRCryptoCreateLocalSignedFDRCertificateData to produce locally-signed FDR certificates indistinguishable from those generated by legitimate Apple restore tools.

The RSA certificate also uses SHA-1 signatures (deprecated) and references fdr-dev@group.apple.com, suggesting a development artifact that persisted into production firmware.

## Severity

High -- Universal cryptographic key material exposure in publicly accessible firmware images.

## Affected Component

libFDR.dylib on iOS/iPadOS restore ramdisks (both restore and update variants).

## Affected Versions

Confirmed on iPadOS 26.0 (build 23A341) for iPad Pro A12X/A12Z (iPad8,10). The same keys are present in both the restore ramdisk (090-88912-697.dmg) and the update ramdisk (090-88917-698.dmg) within the same IPSW. Given the static nature of the keys, all iOS/iPadOS versions shipping libFDR.dylib in restore ramdisks are likely affected.

## Reproduction Steps

### Prerequisites

Any macOS computer with standard command-line tools. No special software, device access, or jailbreak required.

### Step 1: Obtain an IPSW

Download any publicly available IPSW from ipsw.me or Apple's update servers. For this report: iPad_Pro_A12X_A12Z_26.0_23A341_Restore.ipsw (9.6 GB).

### Step 2: Extract the restore ramdisk

IPSW files are zip archives. The restore ramdisk is an IMG4P-wrapped APFS image.

```bash
mkdir extracted && cd extracted
unzip ../iPad_Pro_A12X_A12Z_26.0_23A341_Restore.ipsw
```

The file 090-88912-697.dmg is the restore ramdisk in IMG4P format. Extract the APFS payload:

```python
with open("090-88912-697.dmg", "rb") as f:
    data = f.read()
    apfs_offset = data.find(b'NXSB') - 0x20
    with open("restore_ramdisk.dmg", "wb") as out:
        out.write(data[apfs_offset:])
```

### Step 3: Mount and extract keys

```bash
sudo hdiutil attach -readonly restore_ramdisk.dmg
# Mounts at /Volumes/ramdisk

# Extract RSA private key
strings /Volumes/ramdisk/usr/lib/libFDR.dylib | \
    awk '/-----BEGIN RSA PRIVATE KEY-----/,/-----END RSA PRIVATE KEY-----/' \
    > fdr_rsa.key

# Extract EC private key
strings /Volumes/ramdisk/usr/lib/libFDR.dylib | \
    awk '/-----BEGIN EC PRIVATE KEY-----/,/-----END EC PRIVATE KEY-----/' \
    > fdr_ec.key

# Extract certificates
strings /Volumes/ramdisk/usr/lib/libFDR.dylib | \
    awk '/-----BEGIN CERTIFICATE-----/,/-----END CERTIFICATE-----/' \
    > fdr_certs.pem
```

### Step 4: Verify extracted keys

```bash
$ openssl rsa -in fdr_rsa.key -check -noout
RSA key ok

$ openssl ec -in fdr_ec.key -check -noout
EC Key valid.

$ openssl x509 -in fdr_certs.pem -subject -issuer -dates -noout
subject=C=US, ST=California, L=Cupertino, O=Apple Inc., OU=CoreOS, CN=FDR-LOCAL-V1, emailAddress=fdr-dev@group.apple.com
issuer=C=US, ST=California, L=Cupertino, O=Apple Inc., OU=CoreOS, CN=FDR-LOCAL-V1, emailAddress=fdr-dev@group.apple.com
notBefore=May  9 15:29:41 2022 GMT
notAfter=May  6 15:29:41 2032 GMT
```

### Step 5: Confirm keys are identical across ramdisks

The IPSW contains two ramdisks (restore and update). Both contain the same libFDR.dylib with identical keys:

```bash
KEY1=$(strings /Volumes/ramdisk/usr/lib/libFDR.dylib | \
    awk '/-----BEGIN RSA PRIVATE KEY-----/,/-----END RSA PRIVATE KEY-----/' | \
    openssl rsa -modulus -noout | md5)

KEY2=$(strings "/Volumes/ramdisk 1/usr/lib/libFDR.dylib" | \
    awk '/-----BEGIN RSA PRIVATE KEY-----/,/-----END RSA PRIVATE KEY-----/' | \
    openssl rsa -modulus -noout | md5)

echo "$KEY1"  # ff2ec7190e75c667ce3b163512bdd605
echo "$KEY2"  # ff2ec7190e75c667ce3b163512bdd605
# Identical.
```

## Extracted Key Details

### RSA Private Key

- Algorithm: RSA 2048-bit
- Certificate CN: FDR-LOCAL-V1
- Certificate OU: CoreOS
- Issuer: Self-signed
- Validity: 2022-05-09 to 2032-05-06 (10 years)
- Signature: sha1WithRSAEncryption (deprecated)
- Contact: fdr-dev@group.apple.com
- Key Usage: Digital Signature, Non Repudiation, Key Encipherment
- Apple OID 1.2.840.113635.100.6.16: permissions
- Apple OID 1.2.840.113635.100.6.17: clientid

### EC Private Key

- Algorithm: ECDSA P-256 (prime256v1)
- Certificate CN: FDR-LOCAL
- Certificate OU: Devices
- Issuer: FDR-CA1-ROOT-LOCAL (California, Apple Inc.)
- Validity: 2022-04-21 to 2032-04-18 (10 years)
- Signature: sha256WithRSAEncryption
- Apple OID 1.2.840.113635.100.6.17: Local

## Functions That Use These Keys

The following exported symbols in libFDR.dylib use the embedded keys:

```
AMFDRCryptoCreateLocalSignedCertificateData
AMFDRCryptoCreateLocalSignedDataSignature
AMFDRCryptoCreateLocalSignedFDRCertificateData
AMFDRCertificateSigningRequest
AMFDRCryptoCreateFDRSignatureInternal
AMFDRDataLocalSign
AMFDRDataLocalCreateFullKey
AMFDRCreateLocalPending
```

These functions are called by restored_update and restored_external during device restore operations to sign FDR data without contacting Apple servers.

## Security Impact

1. Universal key compromise. The same RSA and EC private keys are embedded in every IPSW for every device model. Extracting keys from one IPSW compromises local FDR signing for all Apple devices. This is a single point of failure across the entire fleet.

2. FDR certificate forgery. An attacker can use AMFDRCryptoCreateLocalSignedCertificateData with the extracted keys to produce locally-signed certificates that the FDR subsystem accepts as legitimate during restore operations.

3. Device identity material. The FDR-LOCAL certificates are referenced during device identity operations (AMFDRCopyCertifyClientIdSik, AMFDRChipCertifyForRestore). The extracted keys could be used to forge device identity assertions.

4. No Secure Enclave protection. The keys are static data in a shared library, not derived from per-device Secure Enclave material. This defeats the purpose of hardware-backed security for FDR operations.

5. Development artifact in production. The fdr-dev@group.apple.com email address and SHA-1 signature algorithm suggest these keys originated in a development environment and were inadvertently shipped in production firmware.

## Suggested Remediation

1. Remove static private keys from firmware images. Local signing operations during restore should use per-device keys derived from the Secure Enclave or provisioned during factory calibration.

2. Rotate the exposed keys immediately. FDR-LOCAL-V1 and FDR-LOCAL certificates and keys should be considered compromised. Issue replacement keys with new certificate chains in the next firmware update.

3. Encrypt the restore ramdisk. Unlike the root filesystem (which uses AEA encryption), the restore ramdisks are unencrypted, making key extraction trivial. Encrypting them with device-class GID keys would prevent extraction without hardware access.

4. Upgrade cryptographic standards. Replace the SHA-1 signed RSA certificate with SHA-256 minimum, per current industry standards.
